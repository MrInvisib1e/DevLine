# DevFlow v3.1 — Execution Overhaul Design

**Date:** 2026-05-04
**Status:** Approved
**Predecessor:** DevFlow v3 (`docs/specs/2026-05-04-devflow-v3-design.md`)

---

## Executive Summary

DevFlow v3.1 addresses six systemic problems discovered during real-world testing of v3: excessive platform permission prompts, per-slice retry loops that burn tokens, missing security review in the feature pipeline, no read-only investigation capability, incomplete cleanup after feature completion, and no graceful handling of token budget exhaustion. The eight design sections collectively restructure Phase 3 from per-slice retry loops to batch-first execution (reducing worst-case agent dispatches from 9N to 7N), add two new skills (`/security` and `/investigate`), configure platform permissions automatically, enhance agent prompts with richer context, add token budget estimation with graceful checkpoints, improve Phase 6 cleanup with auto-archival and changelog generation, and add a platform agent compatibility layer for environments without subagent support.

## Section 1: Platform Permission Configuration

### Problem

Claude Code, OpenCode, and Gemini CLI prompt "allow bash?", "allow file access?" for every DevFlow shell command. `df-install` creates plugin symlinks, CLAUDE.md, slash commands, and hooks — but does NOT configure platform-level tool permissions. Users experience 10-20+ permission prompts per session.

### Solution

`df-install` configures platform permission allowlists during installation. Merge-only — never removes existing permissions.

---

### Claude Code

Add/merge `.claude/settings.json`:

```json
{
  "permissions": {
    "allow": [
      "Bash(df-*)",
      "Bash(devflow*)",
      "Bash(git *)",
      "Bash(jq *)",
      "Bash(sqlite3 *)"
    ]
  }
}
```

**Merge logic:** Read existing file → parse JSON → append to `permissions.allow` array (deduplicate) → write back. If file doesn't exist, create it.

---

### OpenCode

Add permissions to plugin config in `opencode.json`:

```json
{
  "permissions": {
    "allow": ["df-*", "devflow*", "git", "jq", "sqlite3"]
  }
}
```

---

### Gemini CLI

Add to `.gemini/settings.json` allowlist:

```json
{
  "tool_permissions": {
    "allowed_commands": ["df-*", "devflow*", "git", "jq", "sqlite3"]
  }
}
```

---

### Cursor

No programmatic permission API. Document in `.cursor-plugin/README.md`:
> "Add `df-*`, `devflow*`, `git`, `jq`, `sqlite3` to your Cursor settings → AI → Allowed Commands."

---

### Autonomy

**T1 Silent** — mechanical, reversible. Permission configs are additive-only and can be manually removed.

---

### Implementation

Modify `bin/df-install` to add a `configure_permissions()` function called after plugin registration. Uses `python3 -c` for JSON merge (already a dependency of df-install).

**Files modified:** `bin/df-install` (+~60 lines)

## Section 2: Phase 3 Batch Execution Rewrite

### Problem

Current Phase 3 (`skills/feature/phases/phase-3-execution.md`) runs a per-slice loop: implement → test → review → retry(3x). This means:
- A single test failure triggers 3 retries before moving to the next slice
- Each retry dispatches a full agent with exploration overhead
- Worst case: 9N agent dispatches for N slices (3 implement + 3 test + 3 review)
- Agents waste tokens re-exploring context that could be provided upfront

### Solution: Batch-First Execution

Replace per-slice retry loops with batch stages. Each stage processes ALL slices in the batch before moving to the next stage. Fix passes operate on failures from the previous stage.

---

### New Execution Flow (Per Batch)

```
Stage 1: IMPLEMENT ALL slices in batch
    ↓
Stage 2: TEST ALL slices
    ↓
Stage 3: BATCH-FIX test failures (max 2 passes)
    ↓
Stage 4: SECURITY REVIEW all passing slices
    ↓
Stage 5: BATCH-FIX security CRITICAL/HIGH (max 2 passes)
    ↓
Stage 6: RE-TEST after security fixes
    ↓
Stage 7: SLICE REVIEW (architecture/conventions)
    ↓
Stage 8: BATCH-FIX review findings (max 1 pass)
    ↓
→ Next batch or Phase 4
```

---

### Agent Dispatch Budget

| Stage | Dispatches | Role Constraint |
|-------|-----------|-----------------|
| 1. Implement | N | Implement only — no tests, no refactoring |
| 2. Test | N | Write tests only — no implementation changes |
| 3. Fix tests | ≤2N | Fix test failures only — no new features |
| 4. Security | N | Review only — no modifications |
| 5. Fix security | ≤2N | Fix CRITICAL/HIGH only — no refactoring |
| 6. Re-test | N | Run tests only — no changes |
| 7. Slice review | N | Review only — no modifications |
| 8. Fix review | ≤N | Fix convention violations only |
| **Worst case** | **7N** | vs current 9N |

**Best case (no failures):** 4N (implement + test + security + review).

---

### Stage Decision Table

| Stage Result | Action |
|-------------|--------|
| All slices pass stage | → proceed to next stage |
| Some slices fail | → collect failures, proceed to fix stage |
| All slices fail | → T2 inform, proceed to fix stage |
| Fix pass 1 resolves all | → proceed to next stage |
| Fix pass 2 still has failures | → mark remaining as STUCK, proceed |
| BLOCKED slice | → skip in ALL subsequent stages |
| All slices STUCK after Stage 3 | → T3 gate (ask user direction) |
| DEFAULT | → proceed to next stage with available slices |

---

### State Tracking

Slice JSON gets a `stages` object:

```json
{
  "id": "slice-1-auth",
  "status": "in_progress",
  "stages": {
    "implement": { "status": "done", "agent_id": "..." },
    "test": { "status": "done", "pass": true },
    "test_fix_1": { "status": "done", "pass": true },
    "security": { "status": "done", "findings": [] },
    "security_fix_1": null,
    "retest": { "status": "done", "pass": true },
    "review": { "status": "done", "verdict": "PASS" },
    "review_fix_1": null
  }
}
```

Resume reads `stages` to determine where to continue.

---

### Quick Mode

Same 8 stages, executed sequentially (no worktrees). N=1 per stage for ≤3 slices.

## Section 3: `/security` Skill

### Problem

No security review exists in the DevFlow pipeline. Code ships without automated vulnerability scanning. AI-only security review has 63% F1 score; hybrid static+LLM achieves 81%.

### Solution

A two-pass AI security review skill with optional static analysis anchoring, integrated into Phase 3 as Stage 4.

---

### Trigger & Scope Modes

| Command | Scope | Input |
|---------|-------|-------|
| `/security` | Changed files (diff) | `git diff --name-only HEAD~1` or PR base |
| `/security all` | Full codebase | All source files, per-file chunked |
| `/security <path>` | Targeted | Specific file or directory |
| Phase 3 Stage 4 | Per-slice | Files modified by slice implementation |

---

### Architecture: 3-Pass Pipeline

**Pass 1 — Discovery (per-file)**

For each file in scope:
1. Detect stack from file extension + `.devflow/config.json`
2. Load stack-specific checklist (OWASP Top 10:2025)
3. Analyze file against checklist categories
4. Generate findings with confidence ≥ 0.7
5. Require `exploit_scenario` field — if AI can't write concrete attacker actions, drop finding

**Pass 2 — Triage (per-finding)**

For each finding from Pass 1:
1. Review finding in isolation
2. Check against exclusion list (16 items)
3. Check against project-specific memories
4. Decision: KEEP or DISCARD
5. Assign final severity: CRITICAL / HIGH / MEDIUM / LOW

**Pass 3 — Fix Suggestions (separate pass, only if requested)**

For each KEEP finding:
1. Generate fix recommendation with code example
2. Note: finding and fixing are NEVER in the same pass (research-validated)

---

### Exclusion List (16 items)

Adapted from Anthropic's `claude-code-security-review`:

| # | Exclusion | Reason |
|---|-----------|--------|
| 1 | Denial of Service / resource exhaustion | Not a code-level vulnerability |
| 2 | Rate limiting concerns | Infrastructure, not code |
| 3 | Memory safety in non-C/C++ code | Language runtime handles this |
| 4 | XSS in React/Svelte without `dangerouslySetInnerHTML`/`{@html}` | Framework auto-escapes |
| 5 | Log spoofing | Not exploitable |
| 6 | SSRF that only controls path, not host/protocol | Limited impact |
| 7 | Regex injection | Not a standard vulnerability class |
| 8 | Prompt injection in reviewed code | Out of scope for code review |
| 9 | Lack of audit logs | Not a vulnerability |
| 10 | Missing hardening measures (defense-in-depth) | Only flag concrete vulns |
| 11 | Theoretical race conditions (no practical exploit) | Noise |
| 12 | Environment variables / CLI flags as trusted | Attacker can't modify |
| 13 | UUIDs assumed unguessable | Standard assumption |
| 14 | Vulnerabilities in test files | Not production code |
| 15 | Markdown/docs files | Not executable |
| 16 | Outdated third-party library versions | Managed separately (Dependabot/Renovate) |

---

### Stack-Specific Patterns

Detected from `.devflow/config.json` `stack.runtime` and `stack.frontend`:

| Stack | Additional Checks |
|-------|------------------|
| .NET/C# | `BinaryFormatter`, `TypeNameHandling.All`, raw ADO.NET, missing `[Authorize]`, `appsettings.json` secrets |
| Node.js/TS | `eval()`, `child_process.exec` with user input, prototype pollution, `.env` in client bundle |
| Python | `pickle.loads(user_input)`, `os.system()`, `cursor.execute(f"...")`, Django `@login_required` missing |
| SvelteKit | `{@html}` with user input, missing CSRF on form actions, exposed server-only data in `+page.ts` |

---

### Project-Specific Memories

Stored in `.devflow/config.json`:

```json
{
  "security": {
    "memories": [
      "All /api/ routes go through auth middleware",
      "UUIDs are used as primary keys — no sequential IDs",
      "File uploads are validated server-side with allowlist"
    ]
  }
}
```

Reduces false positives by ~40% (Semgrep research). Memories are checked during Pass 2 triage.

---

### Optional Static Analysis Integration

If `semgrep` is on PATH:

```bash
semgrep --config=auto --json --quiet <files>
```

Parse JSON output → merge with AI findings → deduplicate by file:line. Hybrid approach achieves 81% F1 vs 63% AI-only (DeepSource benchmark).

## Section 4: `/investigate` Skill

### Problem

DevFlow has `/fix` (modify code) and `/review` (review diff) but no read-only investigation capability. Developers need to understand business flows ("what happens when user does X?") and investigate problems ("why does X break?") without modifying code.

### Solution

A read-only investigation skill combining business flow tracing and problem investigation, grounded in the DevFlow knowledge graph.

---

### Trigger & Scope Modes

| Command | Mode | Purpose |
|---------|------|---------|
| `/investigate "what happens when user publishes a story"` | Flow trace | Trace execution path end-to-end |
| `/investigate "why does the comment endpoint return 500"` | Problem investigation | Form hypotheses, gather evidence |
| `/investigate auth-service` | Node-focused | Deep dive on a specific graph node |

---

### Architecture: 5-Phase Pipeline

**Phase 1 — Pre-Investigation Audit (T1)**

1. Parse query → determine mode (flow trace / problem / node-focused)
2. Run `df-explain --rank` → identify relevant nodes
3. For each candidate node: state what is KNOWN (from graph) vs ASSUMED
4. Output: file list with certainty annotations

**Phase 2 — Boundary Identification (T1)**

1. Identify system boundaries for the flow:
   - **Inputs:** HTTP request, event, queue message, CLI command
   - **Outputs:** HTTP response, DB write, event published, email sent
   - **Persistence:** DB tables, cache keys, file writes
2. Output: boundary map

**Phase 3 — Top-Down Trace (T2)**

1. Start from entry point (route handler, event listener)
2. Follow call graph through each layer
3. At each step, apply **dead reckoning guard:**
   - What was directly OBSERVED in code
   - What is INFERRED (and from what evidence)
   - What NEEDS VERIFICATION before proceeding
4. Use **quote-first pattern:** cite code in `<quotes>` before reasoning in `<analysis>`
5. Stop and note uncertainty rather than inventing plausible answers

**Phase 4 — Bottom-Up Verification (T1)**

1. For each component identified in Phase 3:
   - Read actual implementation
   - Verify behavior matches Phase 3 trace
   - Note discrepancies
2. Output: verified trace with corrections

**Phase 5 — Synthesis (T2)**

1. Compile structured report
2. Add uncertainties section (what wasn't examined)
3. Add key invariants (business rules enforced at each step)
4. For problem investigation mode: output ranked hypotheses with evidence

---

### Problem Investigation Mode

Uses **hypothesis tree pattern:**

1. Form 2-3 competing hypotheses about the problem
2. For each hypothesis:
   - List evidence FOR
   - List evidence AGAINST
   - Assign confidence (0.0-1.0)
3. Read files to gather evidence
4. Update hypothesis tree after each file
5. Do NOT converge until all relevant files read
6. Output: ranked hypotheses with evidence, NOT a fix

**Key difference from `/fix`:** NEVER modifies code. Unlimited hypothesis cycles (no 3-cycle cap). Output is report only.

---

### Output Structure

Location: `.devflow/investigations/<date>-<slug>.md`

```markdown
# Investigation: [Query]
**Date:** YYYY-MM-DD | **Mode:** flow-trace | **Duration:** ~3 min

## Entry Point
- **File:** src/Controllers/StoryController.cs:142
- **Trigger:** POST /api/stories/{id}/publish

## Step-by-Step Trace
1. **Validation** — StoryController.cs:156
   - PublishRequestValidator checks WordCount > 0
   - Missing: no check for story ownership (handled by policy)

2. **Authorization** — StoryController.cs:148
   - [Authorize] attribute + StoryOwnerPolicy
   - Policy reads UserId from JWT claims

3. **Business Logic** — StoryService.cs:89
   - Sets Status = Published, PublishedAt = DateTime.UtcNow
   - Raises StoryPublishedEvent via MediatR

4. **Persistence** — StoryRepository.cs:45
   - EF Core UPDATE stories SET status, published_at
   - Single transaction, no explicit SaveChanges boundary

5. **Side Effects**
   - NotificationService handles StoryPublishedEvent → emails followers
   - SearchIndexer handles event → updates Elasticsearch index

## Uncertainties
- NotificationService.cs was not examined — email template unknown
- SearchIndexer retry behavior on Elasticsearch failure unknown

## Key Invariants
- Only story owner can publish (enforced by StoryOwnerPolicy)
- PublishedAt is server-set, never client-supplied
- Story must have WordCount > 0 to publish

## Security Notes
- Owner check happens before load — no IDOR risk
- JWT claims are the sole identity source
```

### Auto-Purge

Configurable in `.devflow/config.json`:

```json
{
  "archive_retention_days": 7
}
```

**Trigger:** Runs when archiving a new plan (not on a schedule).

**Logic:**
```bash
find .devflow/plans/archive/ -maxdepth 1 -type d -mtime +${retention_days} -exec rm -rf {} \;
```

**Default:** 7 days. Set to `0` to disable purging.

---

### Lifecycle Rules

| Artifact Type | Location | Archived? | Purged? |
|--------------|----------|-----------|---------|
| Plan + slice files | `.devflow/plans/<slug>/` | Yes → `archive/` | Yes (after retention) |
| Active symlink | `.devflow/active` | Removed | N/A |
| Security reports | `.devflow/security/` | No | No (manual cleanup) |
| Investigation reports | `.devflow/investigations/` | No | No (manual cleanup) |
| CHANGELOG.md | `.devflow/plans/archive/<slug>/` | Part of archive | Yes (with archive) |

## Section 6: Platform Agent Compatibility Layer

### Problem

Phase 3 dispatches subagents via the `Task` tool (Claude Code) or `task` tool (OpenCode). Gemini CLI and Cursor may not support subagent dispatch. Features fail silently on unsupported platforms.

### Solution

Detect platform capabilities at Phase 3 start. Use subagents where available, fall back to sequential inline execution elsewhere.

---

### Detection (T1)

At Phase 3 start, check for subagent capability:

| Platform | Detection Method | Dispatch Mode |
|----------|-----------------|---------------|
| Claude Code | `Task` tool available | `subagent` (parallel) |
| OpenCode | `task` tool available | `subagent` (parallel) |
| Gemini CLI | No `Task`/`task` tool | `inline` (sequential) |
| Cursor | No `Task`/`task` tool | `inline` (sequential) |

---

### Configuration

`.devflow/config.json`:

```json
{
  "agent_dispatch": "auto"
}
```

| Value | Behavior |
|-------|----------|
| `auto` | Detect platform, choose best mode |
| `subagent` | Force subagent dispatch (fail if unavailable) |
| `inline` | Force inline execution |

---

### Subagent Mode (Claude Code + OpenCode)

Current behavior — dispatch one agent per slice per stage. Parallel within batch.

---

### Inline Mode (Gemini CLI + Cursor)

Same prompt templates, wrapped in role-switching blocks:

```
--- ENTERING AGENT MODE: implementation-agent ---
[Full prompt template content with filled slots]
--- AGENT OUTPUT CONTRACT ---
Respond with EXACTLY this format:
STATUS: DONE|BLOCKED
FILES_MODIFIED: [list]
SUMMARY: [one paragraph]
CONCERNS: [list or "none"]
--- END AGENT MODE ---
```

**Key rules:**
- Sequential execution (one slice at a time)
- Same output contract enforced
- Same validation pipeline runs on output
- No worktrees (inline mode is always sequential)

## Section 7: Token Budget Estimation & Graceful Checkpoint

### Problem

Long feature flows exhaust the context window. The AI hits token limits mid-execution with no saved progress, forcing the user to start over. Existing `/feature resume` reads slice JSONs but doesn't know about batch stage progress.

### Solution

Phase-level token estimation with graceful checkpoint when approaching limits.

---

### Estimation Heuristics (T2)

Before each phase, estimate token cost:

| Operation | Tokens per Unit | Unit |
|-----------|----------------|------|
| Implementation | 3,000–8,000 | per slice |
| Test generation | 2,000–5,000 | per slice |
| Test fix | 2,000–5,000 | per failing slice |
| Security review | 1,000–3,000 | per file |
| Security fix | 2,000–5,000 | per finding |
| Slice review | 2,000–4,000 | per slice |
| Review fix | 2,000–5,000 | per slice with findings |

**Phase estimate formula:**
```
phase_cost = sum(operation_cost × unit_count) for each stage in phase
```

Print estimate as T2 inform:
```
[DevFlow] Phase 3 estimate: ~45K tokens for 5 slices (8 stages).
          Budget remaining: ~80K tokens. Proceeding.
```

---

### Checkpoint Trigger

**Proxy metric:** Agent dispatch count (since actual token counting is platform-dependent).

| Threshold | Action |
|-----------|--------|
| N dispatches (default: 15) | T2 warn: "Approaching token budget. N dispatches used." |
| 2N dispatches (default: 30) | Checkpoint: save progress, tell user to resume |
| Remaining work > 80% of remaining budget estimate | Checkpoint: save progress |

---

### Checkpoint Save

When checkpoint triggers:

1. Write current batch/stage progress to all slice JSONs
2. Write checkpoint marker to `plan.md`:
   ```markdown
   ## Checkpoint — YYYY-MM-DD HH:MM
   **Reason:** Token budget approaching limit
   **Completed:** Batch 1 (all stages), Batch 2 (stages 1-3)
   **Remaining:** Batch 2 (stages 4-8), Batch 3 (all stages)
   **Resume command:** `/feature resume`
   ```
3. Print to user:
   ```
   [DevFlow] Token budget checkpoint. Progress saved.
   Run `/feature resume` in a new session to continue from Batch 2, Stage 4.
   ```

---

### Resume Enhancement

Update `skills/feature/phases/resume.md` to:

1. Read `stages` object from each slice JSON
2. Determine last completed stage per slice
3. Determine last completed batch
4. Resume from exact batch + stage position
5. Print T2 summary of what was completed and what remains

## Section 8: Richer Agent Prompts

### Problem

Current agent prompts (`skills/feature/agents/prompts/`) use `{{CONTEXT}}` slot that provides file paths but not file contents. Agents waste tokens exploring the codebase to find patterns, conventions, and interface contracts that the orchestrator already knows.

### Solution

Enhanced prompt slots with exact file contents, pattern examples, and stack-specific rules. Goal: agents never need to explore.

---

### Enhanced Slot Definitions

| Slot | Content | Token Budget |
|------|---------|-------------|
| `{{ROLE}}` | Agent identity + single-sentence mission | ~50 tokens |
| `{{MISSION}}` | What to accomplish (from slice MD) | ~100 tokens |
| `{{SCOPE}}` | Exact file allowlist + prohibitions | ~100 tokens |
| `{{CONTEXT}}` | **Enhanced:** exact file contents for files in scope + interface contracts for adjacent files | ~2,000 tokens max |
| `{{PATTERNS}}` | **New:** 2-3 actual code snippets from codebase showing patterns to follow | ~1,000 tokens max |
| `{{STACK_RULES}}` | **New:** runtime, ORM, frontend, test framework, naming conventions from memory.md | ~200 tokens |
| `{{PRIOR_WORK}}` | Only on retry: last attempt + structured feedback | ~500 tokens |
| `{{OUTPUT_CONTRACT}}` | Exact output format required | ~100 tokens |

**Total budget:** 4,000–6,000 tokens per agent prompt (vs current ~1,500–2,000).

---

### Context Assembly (Orchestrator Responsibility)

Before dispatching each agent, the orchestrator:

1. **Read slice MD** → extract file list, goal, dependencies
2. **Read file contents** for files in scope → truncate to 2K tokens total
3. **Extract patterns** from reference feature (identified in Phase 1 domain analysis):
   - Find 2-3 files that follow the same pattern the slice should follow
   - Extract relevant code blocks (class structure, method signatures, test patterns)
   - Truncate to 1K tokens
4. **Extract stack rules** from `.devflow/active/memory.md`:
   - Naming conventions
   - Layer structure (controller → service → repository)
   - Test framework and assertion style
   - ORM patterns (EF Core vs Dapper)
   - Truncate to 200 tokens
5. **Assemble prompt** by filling all slots in template

## Risk Assessment

### Item 1 — Platform Permissions

| Risk | Probability | Impact | Mitigation |
|------|------------|--------|-----------|
| Permission format changes across platform versions | Medium | Medium | Version-check before writing; thin JSON merge |
| Overly broad permissions (security concern) | Low | Medium | Allowlist is specific to DevFlow commands only |
| Merge corrupts existing settings | Low | High | Read → parse → merge → write; backup before write |

### Item 2 — Phase 3 Batch Execution

| Risk | Probability | Impact | Mitigation |
|------|------------|--------|-----------|
| Batch-fix agents can't fix without per-slice context | Medium | High | Each fix agent gets full slice context + failure details |
| Stage 4 (security) adds latency to every feature | Medium | Medium | Security review is per-file, parallelizable; skip if no source files changed |
| Resume from mid-batch is complex | Medium | Medium | `stages` object provides exact position; tested in resume.md |
| 8 stages feels heavyweight for small features | Low | Low | Quick mode runs same stages sequentially; stages with 0 failures are instant |

### Item 3 — /security Skill

| Risk | Probability | Impact | Mitigation |
|------|------------|--------|-----------|
| False positives erode trust | High | High | 16-item exclusion list + project memories + 0.7 confidence threshold + exploit_scenario requirement |
| False negatives give false confidence | Medium | High | Disclaimer always appended; hybrid with semgrep if available |
| Stack detection wrong → wrong checklist | Low | Medium | Falls back to generic OWASP checks |
| AI can't write exploit_scenario for real vulns | Medium | Medium | Drop to MEDIUM severity instead of discarding entirely |

### Item 4 — /investigate Skill

| Risk | Probability | Impact | Mitigation |
|------|------------|--------|-----------|
| AI hallucinates code behavior | High | High | Quote-first pattern + dead reckoning guard + uncertainties section |
| Investigation takes too many tokens | Medium | Medium | Phase 1 pre-audit limits scope; token budget applies |
| Overlaps with /fix confuses users | Low | Low | Clear distinction: /investigate = read-only report, /fix = modify code |

### Item 5 — Phase 6 Cleanup

| Risk | Probability | Impact | Mitigation |
|------|------------|--------|-----------|
| Auto-purge deletes needed archives | Low | Medium | 7-day default; configurable; CHANGELOG.md preserves summary |
| Changelog generation fails mid-write | Low | Low | Non-blocking; feature completion proceeds without changelog |

### Item 6 — Platform Agent Compat

| Risk | Probability | Impact | Mitigation |
|------|------------|--------|-----------|
| Inline mode output doesn't match contract | Medium | High | Same validation pipeline runs on inline output |
| Platform detection wrong | Low | Medium | `agent_dispatch` config override available |
| Inline mode much slower than subagent | High | Low | Expected; documented as trade-off |

### Item 7 — Token Budget

| Risk | Probability | Impact | Mitigation |
|------|------------|--------|-----------|
| Estimation heuristics wildly wrong | Medium | Medium | Heuristics are conservative (high estimates); dispatch count is backup metric |
| Checkpoint triggers too early | Medium | Low | Thresholds are configurable; default 15/30 dispatches |
| Resume from checkpoint loses context | Low | High | All state in slice JSONs + plan.md; resume reads artifacts, not conversation |

### Item 8 — Richer Agent Prompts

| Risk | Probability | Impact | Mitigation |
|------|------------|--------|-----------|
| 4-6K prompt too large for some models | Medium | Medium | Token budgets per slot are caps, not targets; smaller repos use less |
| Pattern extraction picks wrong examples | Medium | Medium | Patterns from Phase 1 reference feature (user-approved in PRD) |
| File contents in prompt become stale during execution | Low | Medium | Contents read fresh before each agent dispatch |
