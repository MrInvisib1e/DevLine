How should the graph be stored?

Three approaches to fitting the graph into the existing .devflow/ structure

A

Single memory.json — nodes[] array

Extend the existing memory.json with a flat nodes[] array. Each node has typed edges + intent inline. One file, one source of truth.

memory.json
├── stack, architecture, conventions (existing)
└── nodes[] ← new
    ├── { name, type, file, depends_on[], used_by[], intent }
    └── ...

Pros

  • No new files
  • df-sync patches one place

Cons

  • File grows large on big codebases
  • Patching one node rewrites the whole array
B

Split — nodes.json + edges.json ⭐ Recommended

Node definitions and relationships live in separate files. df-sync patches only the file it needs. df-explain loads both and joins them.

.devflow/active/
├── memory.json (stack, arch, conventions)
├── nodes.json ← { id, name, type, file, intent, confidence }
└── edges.json ← { from, to, rel, intent, last_seen_sha }

Pros

  • Add edge without touching node definitions
  • Edges get staleness tracking independently
  • Scales to large codebases

Cons

  • Two files to keep consistent
C

Node-per-file directory

Each node gets its own JSON file: nodes/CommentEntity.json. Surgical updates — change one node, write one file.

.devflow/active/
├── memory.json
└── nodes/
    ├── CommentEntity.json
    ├── CommentService.json
    └── ...

Pros

  • Most surgical patching
  • Easy to delete a node

Cons

  • Many small files
  • Graph traversal requires loading many files
  • Overkill for most repos
