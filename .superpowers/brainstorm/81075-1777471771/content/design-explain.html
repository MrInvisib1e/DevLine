Design: df-explain Impact Analysis

Section 3 of 3 — how df-explain traverses the graph to answer "what breaks if I change X?"

Traversal algorithm
df-explain Comment

# 1. Resolve input to node id
  match "Comment" in nodes.json
  → found: Comment [entity]

# 2. BFS — outbound edges (what Comment depends on)
  edges where from == "Comment"
  → depends_on: User, Story
  → persisted_in: comments

# 3. BFS — inbound edges (what depends on Comment)
  edges where to == "Comment"
  → CommentService uses Comment
  → CommentController uses Comment
  → CommentCreatedEvent depends_on Comment

# 4. Flag stale nodes in result
  check last_seen_sha per node

# 5. Output structured report
df-explain Comment — output
[Comment] entity — Soft-deletable — hide, never purge
file: Entities/Comment.cs
confidence: high

DEPENDS ON (2)
  → User [entity] — author ownership
  → Story [entity] — parent context

DEPENDED ON BY (3) — changing Comment affects these:
  ← CommentService [service] — CRUD operations
  ← CommentController [route] — exposes POST /api/comments
  ← CommentCreatedEvent [contract] — carries comment payload

PERSISTED IN
  → comments table — soft-delete via is_hidden flag

No stale entries detected.

[DevFlow] 3 nodes depend on Comment.
Changing its shape will affect CommentService,
CommentController, and CommentCreatedEvent.
df-explain — three input forms
df-explain Comment
Match by node name. Exact or fuzzy — "Comm" finds Comment, CommentService.
df-explain Entities/Comment.cs
Match by file path. Useful from editor — pass the open file directly.
df-explain --depth 2 Comment
BFS depth flag. Default 1 (direct neighbours). Depth 2 shows neighbours-of-neighbours.
